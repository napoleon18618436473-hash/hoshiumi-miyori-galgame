# 工作流

## 默认执行顺序

1. 明确本轮目标：新增章节、扩展路线、补角色演出、生成素材、修 bug 或补测试。
2. 检查当前代码和文档：`README.md`、`AGENTS.md`、`src/game/*`、`e2e/*`。
3. 拆分 subAgent：
   - 剧情、人物、选择由 Game Design 处理。
   - 角色立绘、背景、UI 和移动端由 Visual/UI 处理。
   - 语音、主题曲、BGM 由 Audio 处理。
   - 即梦视频由 Video 处理。
   - 状态机和组件由 Implementation 处理。
   - 媒体、单测、e2e、截图由 QA 处理。
4. 主会话整合并解决冲突。
5. 跑检查：`npm run test`、`npm run lint`、`npm run build`。
6. 涉及玩家流程时跑 `npm run e2e`。
7. 更新 README、AGENTS 或本文件。

## 新增章节流程

1. 写章节目标、情绪曲线和结尾钩子。
2. 设计场景清单、登场角色、BGM 氛围、关键选择和路线影响。
3. 给每句正式对白分配唯一 `id` 和 `audio` 路径。
4. 在 `src/game/content.ts` 增加场景、对白、选择和视频事件。
5. 在 `src/game/store.ts` 增加必要状态动作。
6. 生成或接入背景、立绘、视频、语音、BGM。
7. 补内容校验、单元测试和 e2e。

## 剧情验收

- 每章必须有清晰钩子、情绪转折和结尾悬念。
- 选择不能只是装饰，必须影响好感、养成数值、flag、routeSeed 或演出。
- 三位女主台词风格必须可区分。
- 恋爱吸引力来自人物关系和选择代价，不使用低俗擦边。

## 视觉验收

- 桌面视口：1440x900
- 笔记本视口：1280x720
- 手机视口：390x844

检查项：

- 背景、立绘、对话框、选择面板层级清楚。
- 当前说话角色高亮，非当前角色压暗。
- 对话框不遮挡人物关键面部。
- 选择按钮文字不溢出。
- 手机端能读、能点、能完成主流程。

## 音频验收

- 每条正式对白都有 `public/assets/voice/*.mp3`。
- 每个场景有独立 `public/assets/bgm/*.mp3`。
- 标题页有主题曲 `public/assets/music/hoshiumi-theme.mp3`。
- 静音按钮必须同时控制 BGM、主题曲、语音和视频声音。
- 新增媒体后用 `ffprobe` 检查可读。

## 视频验收

- 特定场景视频放在 `public/assets/video/`。
- 生成任务 prompt 和结果保留在 `output/` 或 provider 默认记录中。
- 视频层必须可关闭；播放结束后回到正确剧情状态。
- provider 失败时必须明确标注，不把占位或旧视频说成正式产物。

## 已知扩展方向

- 第二章按 `routeSeed` 拆成遥、澪、凛花三条开局。
- 增加存档槽、已读跳过、自动播放、历史回看和语音重播。
- 为每位角色增加表情差分。
- 增加 OP 视频或片头动画。
